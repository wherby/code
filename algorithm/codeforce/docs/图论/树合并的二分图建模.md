这道题是一个非常经典的**二分图/图论建模与组合计数**问题。我们可以通过将“人”与“用户名”之间的选择关系转化为图的连通块结构，来完美地破解它。

下面我们一步步进行深度剖析：

---

## 一、 核心建模：将选择转化为“图”

由于每个人 $i$ 必须在两个用户名 $x_i$ 和 $y_i$ 中**二选一**，且所有人选定的用户名**互不相同**。

我们可以建立这样一个无向图：

* **节点（Vertices）**：所有的用户名（所有出现的字符串/离散化后的标号）。
* **边（Edges）**：每个人 $i$ 相当于在 $x_i$ 和 $y_i$ 之间连了一条**无向边**。
* **问题的本质**：对于前 $k$ 个人（即前 $k$ 条边），我们需要为每条边选择它的一个端点作为“被占用”的用户名，且每条边选出的端点互不相同。这在图论中被称为**基环树/森林的方向定向问题**（或每个边对应一个独立顶点的匹配）。

---

## 二、 连通块的拓扑形态与方案数

当我们只考虑前 $k$ 条边组成的子图时，这个图被拆分成了若干个独立的连通块。对于任意一个连通块，假设它包含 $V$ 个点（用户名）和 $E$ 个边（人）：

由于每个人必须分到一个独特的用户名，这意味着**边的数量（需要分配的人数）不能超过点的数量（可用的用户名数）**，即 $E \le V$。

根据图的性质，$E$ 和 $V$ 的关系决定了该连通块的形态，也决定了方案数：

### 1. 形态一：$E > V$

一个连通块的边数大于点数。这意味着人比用户名还多，根据鸽巢原理，一定会有至少两个人被迫抢同一个用户名。

* **方案数 = 0**。只要前 $k$ 条边中有一个连通块满足 $E > V$，整个前 $k$ 步的方案数直接归零。

### 2. 形态二：$E = V$（基环树 / 环）

这种连通块被称为**基环树（Pseudo-tree）**（即包含恰好一个环的连通图）。

* 此时每个人都要分到一个不同的用户名，说明所有的点都必须被精准占用。
* **方案数 = 2**。因为只要把环上的边顺时针定向或逆时针定向，就能带动整棵树的定向。
* *特例*：如果这个环是一条**自环**（即某人想选的两个用户名居然是一样的，虽然题目说了 $x_i \neq y_i$，但如果两个不同的人连了相同的两个点形成二元环，也属于环），或者环无法定向，方案数需要细分。在这个标准的无向图里，每个基环树连通块的合法定向方案数固定为 **2**（顺时针或逆时针）。

### 3. 形态三：$E = V - 1$（树）

这种连通块是一棵普通的**树（Tree）**。

* 边数比点数少 1，意味着有 $V$ 个用户名，但只有 $V-1$ 个人。所以最终会**留下恰好一个用户名没有人用**。
* **方案数 = $V$**。因为树上的任意一个点被选为那个“没人用的空位”时，整棵树从这个空位作为根节点向外辐射，所有边的定向（每个人的用户名选择）都会被**唯一确定**。

---

## 三、 整体方案数的乘法原理

由于各个连通块之间是完全独立的，前 $k$ 人（边）的合法总方案数，就是**所有连通块方案数的乘积**。

假设当前前 $k$ 条边构成的图里：

* 有 $C_{tree}$ 个连通块是“树”，各个树的点数分别为 $V_1, V_2, \dots$
* 有 $C_{pseudo}$ 个连通块是“基环树”
* 没有满足 $E > V$ 的连通块

那么，前 $k$ 个人的总方案数为：


$$\text{Ans}_k = 2^{C_{pseudo}} \times \prod_{i=1}^{C_{tree}} V_i$$

如果期间出现了任何一个连通块满足 $E > V$（通常是因为某棵基环树又被加了一条边，变成了多环图），则：


$$\text{Ans}_k = 0$$

---

## 四、 动态维护：并查集（Disjoint Set Union）

由于题目要求我们求出从 $k=1$ 到 $n$ 的**每一个 $k$** 的答案，我们不能每次都重新扫一遍图，必须采用**动态加边**的思路。每多加入一个人，相当于在图里加了一条边，我们可以用**并查集**来实时维护每个连通块的状态。

对于并查集的每个集合（连通块），我们需要维护三个信息：

1. `V`：该连通块的顶点数（初始每个单独的用户名 `V = 1`）。
2. `E`：该连通块的边数（初始 `E = 0`）。
3. `is_valid`：该连通块是否依然合法（即 $E \le V$ 是否成立）。

### 算法执行流程：

1. **离散化**：把所有出现的字符串用户名映射为 $1$ 到 $M$ 的整数。
2. **全局状态初始化**：
* 初始化并查集，每个点的 `V = 1, E = 0`。
* 全局维护当前的答案 `current_ans = 1`（可以先不乘树的 $V$，或者把所有孤立点的 $V=1$ 乘进去）。
* 维护当前“不合法连通块（$E>V$）”的数量 `bad_components = 0`。


3. **依次加入第 $k$ 条边 $(u, v)$**：
* **情况 A：$u$ 和 $v$ 属于不同的连通块（树 + 树 合并）**
* 原来两个块的方案数分别是 $V_u$ 和 $V_v$，从全局答案中除掉它们：`current_ans = current_ans * inv(V_u) * inv(V_v)`。
* 合并两个连通块，新块的 `V_new = V_u + V_v`，`E_new = E_u + E_v + 1`。
* 因为新块的总边数 $E_{new} = (V_u-1) + (V_v-1) + 1 = V_u + V_v - 1 = V_{new} - 1$，它**依然是一棵树**！
* 把新树的方案数乘回全局答案：`current_ans = current_ans * V_new`。


* **情况 B：$u$ 和 $v$ 已经属于同一个连通块（树 变成 基环树，或者 基环树 变成 不合法图）**
* 找到这个连通块的根节点，令其 `E += 1`。
* **如果是树变成基环树**（原本 $E = V-1$，现在 $E = V$）：
* 它原来是树，贡献是 $V$，把它从全局答案中除掉：`current_ans = current_ans * inv(V)`。
* 现在它变成了基环树，贡献变成了 $2$，乘回全局答案：`current_ans = current_ans * 2`。


* **如果是基环树变成不合法图**（原本 $E = V$，现在 $E = V+1$）：
* 原本贡献是 2，除掉它。由于它现在不合法了，`bad_components += 1`。






4. **输出答案**：
* 如果 `bad_components > 0`，说明有连通块炸了，当前 $k$ 的答案直接输出 `0`。
* 否则，输出 `current_ans % 998244353`。



通过这种并查集动态维护，我们在加入每条边时只需要 $O(\alpha(M))$ 的常数级时间更新全局乘积。总时间复杂度为 $O(N \log N)$（瓶颈在字符串离散化）或 $O(N \alpha(M))$，能够极其优雅且高效地解决这道递推计数题。

## 🛠️ 5 种加边状态转移详解

假设当前加入的边是 $(u, v)$，通过并查集找到它们所在的连通块根节点分别为 $R_u$ 和 $R_v$。
同时，我们在并查集中为每个根节点维护两个属性：

* `sz[R]`：当前连通块的**点数**（即提示中的“大小”）。
* `has_cycle[R]`：布尔值，记录当前连通块**是否有环**（初始为 `False`）。

全局维护一个变量 `ans`，代表当前所有合法连通块的方案数乘积。

---

### 场景 1：将两棵树相连（$R_u \neq R_v$，且双方都无环）

* **拓扑改变**：两棵树通过一条边连成了一棵更大的新树。
* **数学转移**：先从全局答案中“除以”两棵树原本各自的大小（乘以它们的逆元），再“乘以”合并后的新大小。
* **代码逻辑**：
```python
ans = ans * inv[sz[R_u]] % MOD * inv[sz[R_v]] % MOD
new_sz = sz[R_u] + sz[R_v]
ans = ans * new_sz % MOD
# 并查集维护
parent[R_u] = R_v
sz[R_v] = new_sz

```



### 场景 2：有环的连通块和树相连（$R_u \neq R_v$，一方有环，一方无环）

* **拓扑改变**：基环树和一棵树连在了一起，由于新边连接了两个不同块，没有产生新环，所以整体变成了一棵**更大的基环树**。
* **数学转移**：基环树原本的方案数是 $2$，合并后依然是基环树，方案数还是 $2$（净贡献不变）。而那棵树原本的方案数是它的“大小”，现在它融入了基环树，失去了独立作为树的贡献。因此全局答案只需要**除以那棵树的原先大小**。
* **代码逻辑**：
```python
tree_root = R_u if not has_cycle[R_u] else R_v
ans = ans * inv[sz[tree_root]] % MOD
# 并查集维护
parent[R_u] = R_v
sz[R_v] += sz[R_u]
has_cycle[R_v] = True  # 合并后的新块有环

```



### 场景 3：将两个有环的连通块相连（$R_u \neq R_v$，且两边都有环）

* **拓扑改变**：两个各自包含一个环的块连在一起，整体边数就变成了 $E = (V_u) + (V_v) + 1 = V_{new} + 1$。边数大于点数，出现了“双环”或多环结构。
* **数学转移**：根据抽屉原理，该块方案数直接归零，导致全局 `ans = 0`。
* **代码逻辑**：
```python
ans = 0
# 后续的所有迭代可以直接返回 0

```



---

### 场景 4：在同一个连通块内连边，且本身是树（$R_u == R_v$，且无环）

* **拓扑改变**：在同一棵树的两个节点间连边，这会**产生该块的第一个环**，使这棵树升级为**基环树**。
* **数学转移**：它原本作为树的贡献是它的“大小”，现在变成了基环树，贡献变成了 $2$。因此从全局答案中**除以树的大小，乘以 2**。
* **代码逻辑**：
```python
ans = ans * inv[sz[R_u]] % MOD * 2 % MOD
has_cycle[R_u] = True  # 打上有环标记

```



### 场景 5：在同一个连通块内连边，且本身已经有环（$R_u == R_v$，且有环）

* **拓扑改变**：在一个已经是基环树的块里乱搞再加一条边，产生了第二个环（边数 > 点数）。
* **数学转移**：方案数直接破产归零，`ans = 0`。
* **代码逻辑**：
```python
ans = 0

```

## 💡 为什么需要预处理逆元？

在取模 $998244353$ 的世界里，我们不能直接做除法（比如 `ans /= size`）。提示中提到“预处理逆元”，是为了让我们在 $O(1)$ 的时间内完成“除法”操作。

我们可以在主程序运行前，通过线性的方式预处理出 $1$ 到 $2N$ 所有数字的费马小定理逆元：

```python
inv = [1] * (2 * n + 5)
for i in range(2, len(inv)):
    # 线性求逆元递推式，或者直接用 pow(i, MOD - 2, MOD)
    inv[i] = (MOD - MOD // i) * inv[MOD % i] % MOD

```

有了这个 `inv` 数组，提示里所有的“除以大小”，在代码里只需要写成 `* inv[size]` 即可，完美契合了动态推进的高效要求！这个提示可以说是给这道题提供了一份标准的、毫无破绽的代码工业级设计蓝图。
