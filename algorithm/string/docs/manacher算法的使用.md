[对称字符串匹配的合并](../../codeforce/技巧/对称字符串的合并与manacher算法的使用.py)

这是一段非常精妙的代码，它利用了 **Manacher 算法**和**差分数组**，在 $O(N)$ 的时间内解决了这个复杂的双字符串回文匹配问题。

我们可以把这个问题拆解为：寻找 $A$ 的前缀和 $B$ 的后缀（以及 $B$ 的前缀和 $A$ 的后缀）如何对称地拼接成回文。

---

### 1. 核心思路：将两个序列“耦合”成一个
代码中最关键的一步是：
```python
for i in range(n):
    s[i] = s1[i] * 26 + s2[i]
```
这里将 $A[i]$ 和 $B[i]$ 压缩成了一个数值。为什么要这么做？
回看题目要求的两个回文串：
* $P_1 = A[1 \dots i] + B[j \dots n]$
* $P_2 = B[1 \dots i] + A[j \dots n]$

如果 $P_1$ 和 $P_2$ 同时是回文，意味着：
1.  **外部匹配**：$A$ 的前缀必须与 $B$ 的后缀镜像相等；同时 $B$ 的前缀必须与 $A$ 的后缀镜像相等。
2.  **内部回文**：在剔除掉外部匹配的部分后，剩下的“中间段”必须本身就是回文。

通过 `s1[i] * 26 + s2[i]`，代码实际上是在同时检查这两条路径。如果新序列 `s` 的某个子段是回文，说明 $A$ 和 $B$ 在该位置对应的元素互为镜像。

---

### 2. Manacher 算法的应用
代码随后在处理过的 `tmp`（插空了 `-1` 的序列，用于处理偶数长度回文）上跑了标准 **Manacher 算法**。

* **`manacher[i]`**：表示以 `i` 为中心的最长回文半径。
* 由于我们合并了 $A$ 和 $B$ 的信息，这个回文半径告诉我们：**以某个点为中心，有多少种长度的“中间段”能同时满足 $P_1$ 和 $P_2$ 都是回文。**

---

### 3. 巧妙的统计：Starting & Ending 数组
为了计算总数，代码使用了**差分数组**（Difference Array）来统计每个位置作为回文开头或结尾的贡献：
```python
starting[i - manacher[i]] += 1
starting[i + 1] -= 1
```
* **`starting[k]`**：表示有多少个回文中心覆盖了左端点 $k$。
* **`ending[k]`**：表示有多少个回文中心覆盖了右端点 $k$。

这实际上是在回答：如果我们确定了切割点，有多少种中间回文段可以与之配合？

---

### 4. 最终遍历与突破口
最后的循环是算法的灵魂：
```python
for i in range(n):
    if s1[i] != s2[n - 1 - i] or s2[i] != s1[n - 1 - i]:
        break
    ans += 1 # 基础匹配（即中间没有额外字符的情况）
    if i < n - 1:
        ans += starting[2 * (i + 1)] # 统计从 i+1 开始的各种长度回文
        ans += ending[2 * (n - i - 2)] # 统计在对称位置结束的各种长度回文
```

**逻辑解析：**
1.  **`break` 机制**：由于 $P_1, P_2$ 的外层必须匹配（$A$ 的头对 $B$ 的尾，$B$ 的头对 $A$ 的尾），一旦在某个位置 $i$ 发现不匹配，更高层级的 $i$ 就不可能成立，直接退出。
2.  **双向统计**：
    * `starting[2 * (i + 1)]` 统计的是：当左侧匹配了 $i$ 个字符时，中间那段回文从 $i+1$ 位置开始向右延伸的所有可能。
    * `ending` 的逻辑对称，统计的是从右侧缩减时的可能。

---

### 5. 复杂度与总结

* **时间复杂度**：$O(N)$。Manacher 是线性的，后续的差分求和与单次遍历也是线性的。
* **空间复杂度**：$O(N)$。使用了若干长度为 $2N$ 的数组。

**这套解法的精髓在于：** 它没有去分别讨论 $P_1$ 和 $P_2$，而是通过**字符合并**将两个字符串的约束压减为一个字符串的约束，再用 **Manacher + 差分** 瞬间完成了原本需要 $O(N^2)$ 甚至 $O(N^3)$ 才能完成的子串统计。
