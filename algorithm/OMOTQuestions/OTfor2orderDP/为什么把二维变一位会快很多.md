# 为什么把变量从2维变一维会快很多

这是一个非常经典的高性能计算（HPC）和算法优化问题。在 Python 这种解释型语言中，将 `dp[k][2]` 这种二维数组（或嵌套列表）降维成两个一维数组 `f0[k]` 和 `f1[k]`，性能提升通常能达到 **30% 到 200%**。

主要原因可以归纳为以下三点：

### 1. 减少了“寻址”的开销（指针跳转）
在 Python 中，列表（List）实际上是一个存放指针的数组。
* **二维列表 `f[j][0]`**：Python 必须先找到 `f` 指向的地址，根据 `j` 偏移找到子列表的地址，再根据 `0` 偏移找到最终的值。这涉及**两次内存寻址**。
* **一维列表 `f0[j]`**：只需要**一次寻址**。

在内层循环跑几万次甚至几百万次时，省掉的那一次寻址操作累积起来的时间非常可观。



### 2. 更好的 CPU 缓存友好性（Cache Locality）
虽然 Python 的对象存储比较散乱，但在底层，一维列表的元素在内存中是**连续存放**的。
* 当 CPU 从内存读取数据时，它不会只读一个字节，而是读入一整块（Cache Line）。
* 如果你访问 `f0[j]`，那么 `f0[j+1]` 很可能已经在 CPU 缓存里了。
* 而对于二维列表，`f[j]` 和 `f[j+1]` 对应的子列表可能被存储在物理内存中完全不同的地方。这会导致频繁的 **Cache Miss**，迫使 CPU 停下来等待数据从慢速的内存（RAM）传输过来。

### 3. 解释器的执行路径更短
Python 是一门动态语言，每一行代码都要经过解释器的“翻译”。
* 执行 `f[j][0]` 时，解释器内部要调用两次 `__getitem__` 魔术方法。
* 执行 `f0[j]` 时，只需要调用一次。
对于 Python 这种每一步都有较大开销的语言，减少操作指令的数量就是最直接的加速手段。

---

### 4. 为什么 `v0 if v0 < v1 else v1` 比 `min(v0, v1)` 快？
你在优化中可能也发现了这一点。这也是类似的逻辑：
* **`min()`** 是一个函数调用。在 Python 中，调用函数涉及创建新的栈帧（Stack Frame）、处理参数传递等昂贵操作。
* **`if else`** 是原生的操作码（Opcode）。解释器可以直接在当前路径下执行判断，速度快得多。

### 总结：你的优化逻辑
你通过将 `dp = [[inf] * 2 for _ in range(k+1)]` 转换为：
1.  **`f0 = [inf] * (k+1)`**
2.  **`f1 = [inf] * (k+1)`**

实际上是把**“深度搜索”**变成了**“平行搜索”**。



**现在的代码状态：**
你现在的 `min_cost` 已经是 $O(NK)$ 算法在 Python 下的**理论极限速度**了。如果你还需要更快的速度，那就只能跳出 DP 范畴，去写那个 $O(N \log N)$ 的“反悔贪心”了（也就是你之前看到的那个带堆和双向链表的解法）。

不过，对于大多数面试和竞赛，现在的 $O(NK)$ 配合剪枝和降维，已经足够通过大部分测试用例了。
